# A0.2 Stock Sufficiency 实验报告

日期：2026-07-23

结论状态：`experimentally validated`

Gate 判决：`inconclusive`

## 1. 实验目的与最终结论

A0.2 在不实现 ToolGapController、custom connector 或 lifecycle runtime 的
前提下，比较：

- S0：stock APC；
- S1：stock APC + native `OffloadingConnector`；

并回答一个受限问题：

> 在受控容量压力下，stock GPU cache 与 stock CPU offload 是否留下满足预注册
> 条件、值得进入 A1 controller-seam 验证的 candidate-addressable gap？

Attempt 2 完成了预注册的 90 次比较运行，90/90 均为
`valid_observation`。聚合器没有触发任何 Stop 或 Continue 条件，因此最终判决为：

```text
inconclusive
valid runs did not satisfy a stable registered Stop or Continue condition
```

这不是无效实验，也不是性能胜利：

- 它证明固定 testbed 中，容量压力能够稳定制造 S0 的完整或部分前缀缺失；
- 在三个预注册 material full-miss cell 中，S1 均以可归因的 native CPU restore
  覆盖缺失前缀；
- 现有 pin 不提供 request-scoped load start/end 区间，因此无法证明 restore
  与 active probe 的传输重叠因果；
- 三个 partial-miss cell 不满足预注册 material 定义，不能在看过结果后将其
  重新解释为 Continue evidence；
- 当前证据不授权进入 A1，也不授权实现 candidate-owned lifecycle runtime。

## 3. Attempt 历史与有效性

### 3.1 Attempt 1：保留但不用于结论

Attempt 1 跑满 90 次，但其前两次运行解析出的 GPU KV capacity 为 3151 blocks，
后续 88 次为 3157 blocks。原因是 runner 记录了每次 engine 自动解析的容量，
却没有把 calibration 的 3151 blocks 冻结回 S0/S1 engine。

这破坏了 cell 间共享同一容量分母的 provenance 契约。Attempt 1 的 raw evidence
和报告均被保留，但不进入科学判决。

### 3.2 Attempt 2：最终有效矩阵

修复后，S0/S1 均显式使用 `num_gpu_blocks_override=3151`，并在每个 run 中检查
resolved capacity。Attempt 2 的结果是：

- 90/90 个 registered ordinal 完成；
- 90/90 为 `valid_observation`；
- 90/90 的 capacity 均为 3151 blocks；
- 每个 bundle 恰好包含 7 个证据文件，共 630 个 matrix JSON；
- 聚合结果可由 raw bundles 独立重算，且与跟踪的 summary JSON 一致。

预算 dry-run a04 因 engine core initialization failure 而无效；固定执行环境后
重新运行的 a05 有效，保守预测为 1.494 GPU-hours，低于冻结的 12 GPU-hours
上限。a04/a05 均保留，不删除失败历史。正式证据不根据后续诊断把 a04 的
`verdict.json` 扩写成它没有记录的具体根因。

## 5. 为什么没有触发 Stop

### Stop 1：S0 在所有压力 cell 都达到完整前缀上限

未触发。六个 target/overload cell 均出现 S0 missing prefix：

- 3 个 cell 为 5/5 full recompute；
- 3 个 cell 为 5/5 partial GPU hit。

因此 stock APC 单独覆盖全部压力工作负载的命题为假。

### Stop 2：所有 full miss 的经济头寸都不超过 θ

未触发。三个 material full-miss cell 的 `Δservice` 分别为：

```text
407.449 ms
411.405 ms
1828.322 ms
```

均明显高于对应的 5 ms screening threshold。该差值包含高压力下
chunked-prefill/decode interleaving 的影响，不能被解释为纯 recompute cost；
但足以排除“已观察 full miss 全部没有实质 service-time 头寸”这一 Stop 分支。

### Stop 3：S1 覆盖所有 miss 且没有方向性 trade-off

未参与判决。虽然三个 material cell 中 S1 都有可归因 CPU restore，但
`transfer_overlap_observable=false`。规格禁止以累计 connector stats 或端到端
timing 冒充 request-scoped load/probe overlap。

### Stop 4：某个 stock policy 在全部压力 cell 同时不劣

未触发。S1 在六个压力 cell 中同时不劣于 foreground 与 probe 的 pair 数依次为：

```text
2/5, 5/5, 5/5, 3/5, 0/5, 4/5
```

它没有在全部六个 cell 达到 4/5；S0 在任何压力 cell 都未达到该全局条件。
因此不能将一个 stock static policy 宣布为所有声明压力 cell 的统一不劣解。

## 6. 为什么没有触发 Continue

### Continue 1：material miss 没有被 S1 可归因 restore 覆盖

未触发。三个 material cell 的 15 个 S0 full-miss pair，在 S1 侧全部观察到
与缺失 prefix 对应的 external cached tokens；`unrestored_pairs=0`。

### Continue 2：至少两个 material cell 出现稳定方向性 trade-off

未参与判决，因为 `transfer_overlap_observable=false`。

即使只把当前端到端方向作为非因果诊断，三个 material cell 中满足“foreground
一侧更快、probe 另一侧更低”的 pair 数也只有：

```text
L=2048/target: 3/5
L=2048/overload: 0/5
L=8192/overload: 2/5
```

均未达到单 cell 4/5，更不满足至少两个 material cell 的注册条件。
`L=16384/target` 虽出现 5/5 方向冲突，但它是 partial-miss cell，不能事后代替
material cell。

### Continue 3：material cell 间出现 foreground 方向反转

未触发。三个 material cell 均为 S1 foreground 更快 5/5：

```text
S1_faster:
  L=2048/target
  L=2048/overload
  L=8192/overload

S0_faster:
  none
```

没有出现两个以上 material cell 之间的稳定方向反转。

## 7. 能说什么，不能说什么

### 可以说

- 在该固定 A10/vLLM/model/HND testbed 上，受控容量压力能够稳定制造 stock
  APC 的完整或部分前缀缺失；
- stock native CPU offload 在三个预注册 material full-miss cell 中均提供了
  可归因 restore；
- 90 个有效 run 没有满足任何预注册 Stop 或 Continue 条件，因此 A0.2 的正式
  结论是 `inconclusive`；
- 该负向/未决结论和完整 raw evidence 被保留。

### 不能说

- ToolGap lifecycle runtime 已实现；
- S1 对真实 agent tool-gap 工作负载更快；
- CPU offload 在生产中优于 recompute 或 retain；
- 当前数据证明 transfer contention 或 QoS 因果；
- partial-miss cell 已证明 candidate-owned controller 有价值；
- A0.2 已授权 A1、CT1、CT2、CT3 或简历性能数字。

本 gate 使用 capacity-pressure barrier，而不是真实 wall-clock tool latency、
arrival process 或时间驱动 LRU。执行是真实 GPU，工作负载仍是受控合成 workload，
不能外推为生产代表性。

## 8. 决策与后续方向

当前 A0.2 应按有效的 `inconclusive` 结果收口，不追加选择性 run，不修改 material
定义，也不直接进入 A1。

正式项目决策记录为
[D028](../../docs/agent-kv/DECISIONS.md#2026-07-23-a02-stock-sufficiency-record)：
关闭当前 A0.2 gate，不从该证据进入 A1。

下一步首先是项目级评审，而不是继续写 runtime：

1. 按 D018/D024 重新判断这条主线是否仍能证明一个可维护、非重复、
   candidate-owned 的 lifecycle contract；
2. 如果停止或收窄，保留 A0.2 作为 stock baseline 与负向 recruiting evidence；
3. 如果继续，必须新建独立 ticket 和新规格，只能提出一个新的可证伪问题，例如：
   - 是否存在可维护的 request-scoped transfer interval seam；
   - partial miss 的 recompute/restore 边界是否值得独立研究；
   - restore 与 active-request QoS 的因果关系能否通过窄观测 seam 证明。
4. 新 gate 必须重新预注册 material 定义、观测 seam、Stop/Continue 条件和成本上限；
   不得用本次数据追认 A0.2 为 pass。

`L=16384/target` 的 5/5 foreground/probe 方向冲突只能用于生成下一轮假设。
它既不是本 gate 的 material cell，也没有 transfer overlap 因果证据。
