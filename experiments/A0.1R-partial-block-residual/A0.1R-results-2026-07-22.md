# A0.1R 执行汇总：partial-block residual 的 stock-APC admission

> 结论状态：`experimentally validated`（固定单机 testbed、固定 canonical pair 的必要前置证据）
>
> 主结论 artifact：本地未纳入 Git 的 `raw/task-0/task0-oNN-a01/`。三份 bundle 的逐文件
> SHA-256 见第 6 节。

## 1. 结论摘要

**A0.1R 的三个预注册独立 ordinal 全部为 `admission_pass`。**

在固定 vLLM、模型、模板、fixture 且无压力的单请求环境中：R0 先生成 tool call 并 warm
prefix；随后 canonical R1 的 `RequestOutput.num_cached_tokens` 在三次独立冷启中均为
`192`，恰好等于 A0.1 已证明的完整块可复用上限 `C=192`。

这回答了一个窄但关键的问题：A0.1 的 `LCP=199`、`floor(LCP/16)*16=192` 并非只是一段
离线 token 算术；stock vLLM APC 确实将这 192 个 token materialize 为 R1 的本地缓存命中。

因此，A0.1 的历史 `serialization_stop` 不应被解释成“token 无法 round-trip”或“stock APC
不能命中”。它仍保留为原 A0.1 **full-block coverage** 判据下的负向结果；A0.1R 不改写该
bundle 或历史 verdict，只额外验证了它留下的 partial-block residual。

## 2. 已回答与未回答的问题

### 已回答

- 固定 canonical R0/R1 pair 下，R0 是冷请求（`num_cached_tokens=0`）；
- R1 的 cache accounting 报出 `192` 个本地 cached tokens，而不是 0、缺失值或一个少于
  `C` 的 block-aligned 数；
- 该结果在三个重新创建的单请求 in-process `LLM` 中一致复现；
- A0.1 的 tool-call semantic token 一致性和 `LCP=199` 与此次引擎观测相容。

### 未回答，且不得推断

- tool gap 在真实 agent 负载中是否会导致这 192 个 token 被驱逐；
- HBM retain、CPU offload、recompute 三条路径的延迟、吞吐、成本或胜负边界；
- 任何 queueing、P99 resume latency、TTFT、goodput 或 ToolGapController 收益；
- 多轮 tool call、其它模型/模板、其它 GPU，或 production agent traffic 的可泛化结论；
- candidate-owned lifecycle runtime 是否可实现、正确，或与 vLLM 的 maintainable seam 是否存在。

所以这是一条 **A0.2 之前的 admission evidence**，不是 A0.2 的 S0 control cell，更不是
性能结果。

每条 run 同时满足：

```text
R0.num_cached_tokens == 0
R1.num_cached_tokens == eligible_full_prefix_tokens == 192
R1 cached tokens <= LCP == 199
semantic_span_equal == true
```

R1 prompt 总长度为 `249`，因此本次 R1 的 prompt accounting 中有 `57` 个 token 不由这个
已命中的完整前缀覆盖。这正符合实验对象：它验证 APC admission，而不是把整个 R1 错称为
cache hit。

## 5. 执行过程与有效性

正式样本之前出现过两次启动阻塞，均发生在 R0 请求之前，raw 目录当时只有 `.gitkeep`，所以
不属于 `invalid_run` 或被排除的 ordinal：

1. runner 把 Hub ID 直接交给 vLLM，vLLM 初始化时重新访问 Hugging Face；云机出口不可达。
   修复为：先离线解析冻结 snapshot，再传入本地 model/tokenizer 路径；
2. FlashInfer 在首次 sampler JIT 时需要 `ninja`，但非交互启动没有把 venv 的 `bin/` 放入
   `PATH`。修复为：启动命令显式包含 `/root/toolgap-venv/bin`。

以上修复及 `num_cached_tokens` 可观测性配置在正式 ordinal 之前提交为 `8d3accd`；三个 bundle
记录的 `project_head` 与 tracked-input SHA-256 完全一致。每份 bundle 都包含且只包含
`manifest.json`、`r0.json`、`r1.json`、`accounting.json`、`verdict.json`。

一个必须保留的运行时告警是：vLLM 对 Qwen2.5-7B 在
`enable_chunked_prefill=False` 下发出“not officially support”的 warning。三次运行均成功完成，
但这不是对该 pin 在任意 workload 下正确性的证明；A0.2 之前必须把它作为配置风险单独处置。

## 7. 对项目状态的影响

### 可更新的事实

- A0.1 的 6-token partial-block residual 是有界的，并未阻止 stock APC 使用其前面的 192
  个完整 token；
- “stock APC 不能 materialize 该 canonical pair 的 full-block prefix”这一解释被直接反证；
- A0.2 的零压力 APC-hit **必要前置**已经成立。

### 仍不可更新的事实

- 项目尚没有 candidate-owned lifecycle runtime；
- 尚没有 APC retention-under-pressure、CPU offload 或 recompute 的受控比较；
- 尚没有性能指标，因此简历与项目叙事中仍不得出现延迟、吞吐或收益数字。

现有 `D026` 和 `docs/agent-kv/README.md` 仍反映 A0.1 的历史停止分支。它们不能被静默改写：
若本报告经评审通过，应新增/更新一条显式决策，记录“新的 A0.1R ticket 已直接验证 stock APC
admission；A0.2 由 stopped 转为可重新评审”，同时保留 A0.1 原始负向 full-block coverage
artifact。

## 8. 建议的后续发展（按顺序）

1. **评审并收口本报告。** 决定是否以新的 decision supersede D026；不删除 A0.1 的历史
   verdict，也不把 A0.1R 写成性能成功。
2. **先处理 chunked-prefill 风险。** 对 pin 后的 vLLM 做一个最小 source/runtime audit：要么
   证明在本实验/A0.2 所需的受控单请求路径下关闭它是安全的，要么把 A0.2 的 scheduler pin
   改为实际受支持的配置，并重新确认 accounting mapping 仍成立。这个结论必须在 A0.2 前冻结。
3. **随后重新评审 A0.2 实施。** A0.2 才开始回答容量压力下 stock APC 是否仍命中，以及 miss
   是否有足够的经济头寸；它仍不实现 retain/offload policy，更不实现 lifecycle runtime。
4. **只在 A0.2 证明存在可归因且有价值的 miss 后，推进 Gate A 的 runtime seam / lifecycle
   ownership。** 这时才有理由把问题从“stock APC 能否命中”推进到“如何由 candidate-owned
   runtime 安全地编排 retain、offload 与 fallback”。

这条顺序避免两个错误：因一个 block 对齐诊断过早终止项目，或因一个无压力 admission pass
过早宣称性能价值。
