# A0.1R Task-0：partial-block residual 的 stock-APC admission

## 目的与原因

A0.1 的最终 raw trace 已证明 assistant tool-call semantic span 完整 round-trip：
`semantic_span_equal=true`，`LCP=199`，span 为 `[178,198)`。历史 verdict
`serialization_stop` 的唯一触发条件是 block size `B=16` 下
`floor(199/16)*16=192 < a_end=198`。这个 6-token 余量是 block 对齐诊断，并不等价于
semantic serialization drift。

本实验不重跑或重标 A0.1。它只验证一个此前未被直接观测的引擎事实：当 R0 已经 warm
这条精确前缀时，stock vLLM APC 会不会在无压力 R1 上 materialize 那 192 个完整可复用
token。如果不会，后续 A0.2 的所有 retention/offload 比较都没有稳定的 stock hit 基线；
如果会，才有资格讨论 A0.2。

它不证明真实 tool-gap 时间分布、capacity pressure、CPU offload、吞吐、TTFT 或任何
ToolGapController 价值。

`a0.1-task0-prefix-anchor.json` 是 source run 的最小 tracked derivative。它保存 R1
`prompt_token_ids[:192]` 的 JSON integer list；canonicalization 必须严格为
`json.dumps(list[int], separators=(',', ':')).encode('utf-8')`。

在每个 ordinal 中，R0 和 R1 token IDs 都必须直接来自 `RequestOutput`：

```text
R0 IDs = R0.prompt_token_ids + R0.outputs[0].token_ids
R1 IDs = R1.prompt_token_ids
```

禁止从可见文本重新 tokenize 来构造任一判决输入。

## 可观测源码契约

`RequestOutput.num_cached_tokens` 是唯一 admission counter。针对上述 vLLM pin，源码
审计锁定以下流：

1. `KVCacheManager.get_computed_blocks()` 的本地命中经
   `Scheduler.schedule()` 的 `num_new_local_computed_tokens` 进入 prefill stats；
2. `PrefillStats.set()` 定义 `num_cached_tokens = local + external`；
3. `OutputProcessor.process_outputs()` 将它传播到 `RequestOutput`；
4. Task-0 不配置 connector，所以 external cached tokens 必须为 0；
5. `Request.prefill_stats` 在这个 pin 无条件创建，`disable_log_stats=True` 只关闭
   aggregate `RequestStateStats`，不关闭上述 output propagation；
6. `LLM.__init__` 接受 `spec_method=None`，`EngineArgs` 接受显式
   `enable_chunked_prefill=False`；runner 在 engine 构造后再次断言 APC 开启、chunked
   prefill 关闭。

## 执行与分流

每个 ordinal 都新建一个单请求、无并发的 in-process `LLM`：先以 greedy R0 完成 tool
call，再立即按本次 parser 输出及动态 `tool_call_id` 构建 canonical R1。runner 同时验证
完整 local template render 的 token IDs 与引擎 prompt IDs 相等，并以 A0.1 已验证的
message-scoped offset-overlap adapter 重新获得两侧 span。

| 条件 | Task-0 verdict | 后续动作 |
| --- | --- | --- |
| R0 非冷、counter 缺失/非法、full-block accounting 自相矛盾、或冻结 anchor 漂移 | `invalid_run` | 保存 bundle；不计入 ordinal 结论 |
| semantic span 不等，或 LCP 未覆盖 R1 span 终点 | `semantic_stop` | 保存 bundle；停止，不把它称为 APC 问题 |
| `R1.num_cached_tokens == C` | `admission_pass` | 此 ordinal 通过 |
| `R1.num_cached_tokens < C` 且 block-aligned | `stock_apc_unavailable` | 保存 bundle，暂停余下 ordinal，回到本节源码 mapping 复核；不可直接用一次短报数宣布项目停止 |

目标是三个 registered ordinal 都 `admission_pass`。无 majority vote、无阈值调参、无修改
fixture 的 padding、无同次 bundle 覆盖。每个 run 保存 `manifest.json`、`r0.json`、
`r1.json`、`accounting.json`、`verdict.json`；结果写入 `raw/task-0/task0-oNN-aMM/`。

## 解释边界

三个通过只说明：在固定 pin、固定 canonical pair、无压力的单请求环境中，stock APC 的
公开 request accounting 与 A0.1 的 192-token 完整块前缀一致。它只是 A0.2 的必要前置，
不是 A0.2 的 S0 control cell，也不会自动放行 CPU tier、pressure barrier 或 runtime
实现。
