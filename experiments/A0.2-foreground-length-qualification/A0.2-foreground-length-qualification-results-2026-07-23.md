# A0.2 前台长度资格实验：结果记录（2026-07-23）

> 证据状态：`experimentally validated`，仅适用于本目录的前台 token round-trip 与 stock APC admission。
>
> 本记录不是 A0.2 容量压力矩阵、CPU offload 对照、延迟/吞吐结果或 ToolGap runtime 结果。

## 1. 要验证的命题

对每个注册前台长度 `L ∈ {2048, 8192, 16384}`，在五个顺序、fresh OS
process 中取得可重复的冷 R0；第五个 process 随后提交 canonical R1。只有当 R1 的
`RequestOutput.num_cached_tokens == L`、完整 block LCP 为 `L`，且 tool-call 语义 span
逐 ID 一致时，才将该 bundle promotion 为后续 A0.2 可消费的 tracked anchor。

固定口径：Qwen/Qwen2.5-7B-Instruct 本地 snapshot
`a09a35458c702b33eeacc393d103063234e8bc28`、vLLM
`752a3a504485790a2e8491cacbb35c137339ad34` / `0.25.1+cu128`、A10、block size 16、HND、
`enable_prefix_caching=True`、`enable_chunked_prefill=True`、`disable_log_stats=False`、
无 connector、无 speculative decoding。

## 4. 此结果没有证明什么

- 没有引入 builder pressure，因此没有证明 GPU prefix miss、LRU eviction 或容量压力代理；
- 没有启动 S1 CPU OffloadingConnector，因此没有 CPU restore 结论；
- 没有采集 queue delay、TTFT、throughput 或 active-probe tail，因此没有性能结论；
- 没有实现 ToolGap lifecycle runtime、controller 或 request-scoped policy。

下一步仍是先将 A0.2 comparative runner 的实现方案评审为可执行代码；只有完成 calibration、
probe/connector preflight 后，才可以运行预注册的 90-run S0/S1 容量压力矩阵。
