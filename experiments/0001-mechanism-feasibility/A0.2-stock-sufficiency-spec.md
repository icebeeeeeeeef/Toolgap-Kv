# Gate A0.2：Stock APC / Native CPU Offload Sufficiency 规格（供 Grill）

> 状态：`roadmap`
>
> 前置条件：A0.1R 的 stock-APC admission、chunked-prefill preflight 与
> `experiments/A0.2-foreground-length-qualification/anchors/` 下三份前台 anchor 已关闭；
> A0.2 comparative matrix 本身仍是 `roadmap`。

## 1. 要回答的唯一问题

在固定 vLLM、模型和 A0.1 canonical tool-call fixture 上，**stock APC（S0）和 stock
APC + 原生 CPU OffloadingConnector（S1）是否已经覆盖了所有预注册的 GPU-prefix miss，
且没有留下可归因的恢复/活跃请求 trade-off？**

本规格不实现、加载或比较 ToolGapController（T）。它只决定 A1 是否还有值得验证的
candidate-owned runtime 问题，不能证明 T 一定会更快。

## 2. 被刻意排除的复杂性

不引入第二模型、agent trace、多工具、动态策略、custom connector、`kv_transfer_params`、
per-request offload cap、cache-salt API、LMCache、二级存储或 vLLM core patch。

不把 wall-clock tool gap 作为此 gate 的扫描维度。为了使每个压力 cell 真正达到声明的
working set，工具结果在一个 **pressure barrier** 后才返回；实际 gap 时长会被记录，但
不与别的 gap 长度比较。提前同时扫描到达率、gap 时长、prefix 长度和 M，会使“是压力、
注入速度还是调度造成的 miss”不可归因。G 的独立边界留给 CT3。

因此 A0.2 证明的是**容量压力代理下**的 stock miss/restore 机制存在且可归因；它不证明
真实工具等待的 wall-clock 时序、正常到达率与 LRU 会产生同一个 miss。任何 Continue 只
允许进入 A1 的 seam 验证，不能被表述为生产 tool-gap 的时间驱动 eviction 已被复现。

## 3. 固定基线与公平性

| 配置 | 设置 | 不允许的附加行为 |
| --- | --- | --- |
| S0 | pinned vLLM + APC；关闭 KV offload | ToolGap 代码、custom connector、request-scoped action 注入 |
| S1 | 与 S0 相同的 engine/model/cache/scheduler 参数 + 原生单 CPU-tier `OffloadingConnector` | ToolGap 代码、custom manager、per-request `max_offload_tokens`、其他 storage tier |

两者必须使用相同的：vLLM commit/patch hash、model/tokenizer/template SHA、tool parser、
GPU KV cache budget、`max_model_len`、block size、prefix-caching 开关、显式 resolved 的
chunked-prefill 开关、关闭的 speculative decoding、采样参数、前台 fixture、背景 payload、
随机 seed 和运行顺序。若 launch command 没有把这些值写入 manifest，则 run 无效。

S1 的 CPU tier 容量固定为不小于 `1.5 × C` 的等价 KV block 容量，其中 `C` 是实际
GPU KV cache block 容量。它大于最高 `1.30 × C` 的工作集，目的是避免把“CPU tier
人为装不下”误判为 native offload 本身不足。若目标机器没有足够 DRAM，本次 gate 不运行；
不能悄悄缩小 S1 tier 后继续做公平比较。

`OffloadingConnector` 是独立的 CPU store/load tier，不是 V0 preemption `swap-space`。

## 4. 压力工作集 M

令：

```text
C = 启动后 vLLM 实际分配的 GPU KV cache 完整 block 容量
W = R0 完成到 R1 提交之间，前台候选前缀与背景请求输入注入的、
    互不共享的完整 prefix-cache block 总数
M = W / C
```

`M` 是 **预注册 prefix working-set demand**，不是物理 GPU 总显存占比，也不是声称已经
驻留在 HBM 的比例。固定 Qwen、KV dtype 和 block layout 后，以完整 block 计数等价于
以 bytes 计数，且与 vLLM 的分配/驱逐对象一致。active probe 在 decode 中新增的少量 KV
不计入 M；它使用固定 output cap，并以实际生成 token/block 数单列记录，不能被伪装为
精确压力轴。

前台完整 prefix block 已包含在 `W` 内，不能在 builder wave 中重复计数。calibration 后每个
cell 的背景目标固定为：

```text
foreground_full_prefix_blocks = L / block_size
builder_target_blocks = floor(M * gpu_capacity_blocks) - foreground_full_prefix_blocks
```

若 `builder_target_blocks <= 0`，该配置为 `invalid_configuration`；不得用零 builder 或跳过
前台 block 的方式伪造低压力 cell。

在比较前可做一次 non-comparative calibration：读取 `C`、实际 block size 和每个 payload
的完整 block 数，并验证 driver 能构造首个完整 block 起不共享的 prompt。calibration 不得
提交 tool-gap resume、不得产生 S0/S1 胜负指标。

calibration 后冻结三档：

| band | M | 用途 |
| --- | ---:| --- |
| low | 0.50 | 验证低压力 stock APC 的正向对照 |
| target | 1.10 | 明确超过容量，首次检验 prefix 回收/下层恢复 |
| overload | 1.30 | 检验继续施压是否变成 queue 或 active-tail 代价 |

每格使用 `floor(M × C)` 个完整 block；第一条 S0/S1 comparative trace 后不得修改
band、增加有利 cell 或改变 payload 长度。

本 gate 不声称定位 LRU 的精确拐点：`0.50` 是低压力正向对照，`1.10/1.30` 是两个明确
超过容量的恢复压力点。`M≈0.9–1.0` 的边际区间与连续边界拟合属于 CT3；不在 A0.2
事后增加该档，避免把 90-run falsification gate 扩张为 boundary sweep。

## 5. 工作负载与 pressure barrier

### 前台 resume

前台使用 A0.1 的同一单工具 canonical history。R0 完成并产生一个真实 parser tool
call 后，driver 不立即返回 tool result，而是进入 pressure barrier；barrier 完成后以
同一 canonical R1 提交 resume。

R0 的隔离态是显式不变式：提交 R0 前以及 R0 finish 前，fresh engine 中不得有 builder、
active probe 或其他用户 request 处于 queued、prefill 或 decode in-flight 状态；driver 必须
记录这两个检查点。只有收到 R0 finish 事件后，才允许注入 builder，随后才允许 active probe。
这样 A0.2 消费的 R0 与 A0.1 reproducibility preflight 具有相同的无并发条件；违反即
`invalid_run`，不能把 batch-dependent generation 漂移误读为 cache miss。

每个 `L` 只能消费
`experiments/A0.2-foreground-length-qualification/anchors/foreground-<L>.json` 中登记的
span、LCP、完整-prefix ceiling 与 prefix hash；即
`anchors/foreground-2048.json`、`anchors/foreground-8192.json`、
`anchors/foreground-16384.json`。每个 comparative run 重新计算该 pair 的 R0/R1 token IDs，
并与对应 anchor 比对。若 canonical assistant tool-call、span、LCP、prefix hash 或 accounting
边界不一致，立即 `invalid_run`；不得把 token 漂移后的低 cached-token 归因给 HBM pressure。
没有三份 admission-pass anchor 时不得开始 comparative matrix，runner 也不得在 matrix 内生成
或修改 foreground padding。

前台稳定 prefix 长度 `L` 固定为现有 Phase 0 三点：

```text
L = [2048, 8192, 16384] tokens
```

这些是完整 block 对齐后的目标长度；实际 prompt token 数、R0/R1 LCP 上限与最后完整
cache block 都写入 run artifact，并与对应 anchor 核验。

### 单一背景请求类

背景都是普通、无工具、相同模型与采样参数的请求。每个 payload 从第一个完整 cache block
起包含确定性的唯一 salt；不依赖 private block API 或 `cache_salt` 参数来假定去重。

它们分两波，但仍是同一种普通请求：

1. **builder wave**：R0 之后注入互异的长 prefix，直到累计完整 block 数达到该 cell 的 W；
   已完成的 prefix 仍作为 APC eviction candidate，故能构造确定性 cache working set。
2. **active probe wave**：W 的最后十六个背景请求在 R1 前紧邻启动，使用固定的 32-token
   completion 上限，使 R1 到达时至少一个普通请求仍在执行；其输入 prefix block 已计入 W。

若 builder 没有达到 W，或 R1 提交时没有 active probe，则该 run 为 `invalid_run`。实际
barrier 耗时、背景提交/完成顺序和 R1 到达时 active probe 数必须保存；它们不是本 gate 的
优化指标。

builder 的 touch ordering 是显式不变式：R0 完成后，driver 不得再次提交或重触碰前台
prefix；builder 的每个新 prefix 只写入一次，且其完成时晚于 R0。该设计刻意让前台成为
较早的 APC candidate，以构造**保守的容量最坏顺序**来检验 stock recovery，而不是声称
发现了真实 workload 的 LRU 排序。artifact 保存所有 submit/finish 顺序与 payload hash；
若引擎没有公开 block-rank 事件，不得把内部精确 LRU rank 描述为已证明。

active probe 的 32-token cap 不自适应。comparative run 前先做十次 non-comparative
probe-timing preflight，仅确定一个固定的 probe-to-R1 lead offset；该 offset 必须令至少
九次有 active probe 存活到 R1 提交。若做不到，停止并修正 workload 规格，不能在比较期间
按 run 增大 output cap。比较期间若 active-probe invalid rate 超过 10%，整个 cell 标为
`inconclusive`，而非选择性补跑。

## 6. 矩阵、次数与顺序

唯一比较矩阵为：

```text
L ∈ {2048, 8192, 16384}
M ∈ {0.50, 1.10, 1.30}
policy ∈ {S0, S1}
```

每个 `(L, M)` 运行五个预注册的配对 AB/BA block；每个 block 都执行一次 S0 和一次 S1，
共 5 次 S0、5 次 S1。每个 policy run 重启 engine；同一配对内的 S0/S1 复用完全相同的
workload nonce、payload hashes 与提交时间表，不同配对才使用新的确定性 nonce。这样既避免
跨 run APC 残留，也不把不同 payload 误当成 policy 差异。AB/BA 的起始顺序在运行前由固定
seed 决定并保存。

这给出 90 个 comparative engine runs；AB/BA is a pair-order attribute; it does not add a
schedule dimension。它是 A0.2 的固定成本上限；不因某个 cell 看起来
有希望而追加 run。non-comparative calibration/preflight 后，根据一次完整 cold start 与
一个代表性 `(L=8192, M=1.10)` dry run 估算总 GPU 时间；若 90 runs 的保守预测超过
12 GPU-hours，比较不得开始，必须先重新评审预算或缩小预注册矩阵。若五对样本不足以稳定
区分方向，结论是 `inconclusive`，不是选择性加跑。

## 7. 观察与原始证据

每次 run 必须保存：

```text
provenance
  engine commit/patch, model/tokenizer/template SHA, hardware/driver/CUDA,
  complete launch command, S0/S1 config, GPU KV C, CPU tier capacity

foreground R0/R1
  per-L qualified anchor, prompt/cached/recomputed token accounting,
  GPU APC hit/miss evidence, output oracle,
  resume queue delay, prefill time, TTFT

S1 physical tier
  CPU store/load hit or miss, bytes, transfer start/end/time, load-failure/error evidence

background active probes
  submit/dispatch/finish timestamps, queue delay, TTFT, completion count,
  actual decode tokens/blocks, decode-active interval; cell-level p95/p99 over
  five runs' probe requests

pressure proof
  target and actual W blocks, C blocks, M, payload hashes, builder order,
  barrier elapsed time, active probes at R1 arrival
```

端到端时间不能单独用作 `gpu_hit`、`cpu_restore` 或 `recompute` 的判据。S1 必须同时有
GPU-miss 与 CPU load bytes/time 证据；S0 的 recompute 需 token accounting 佐证。

active-probe trade-off 的预注册机制是：S1 的 H2D restore 与普通请求 decode 争用
PCIe/SM 或由 CPU load 延后 R1 prefill。只有 restore 区间与 probe decode-active interval
存在可记录的时间重叠，才能把“R1 变快而 probe 变慢”归因到该机制；若该重叠所需事件在
pinned seam 不可见，任何 tail 差异只能是 `inconclusive`，不能作为 Continue 证据。

90 个 comparative run 前，必须单独完成 connector-observability preflight：在 pinned
`OffloadingConnector` seam 中，为一次 R1 CPU load 取得可与该 request 关联的单次 load
start/end monotonic timestamps（或同等精度、可构成区间的事件对）。仅有 engine-wide 聚合
bytes/time 不够。若此能力不存在，manifest 必须记为 `transfer_overlap_observable=false`，
Continue 条件 2（directional trade-off）不可用；不得用端到端时间替代。条件 1 与 3 仍可
独立判定，但报告不得把 trade-off 路径叙述为本 gate 的主要证据。

## 8. 预注册判决

本 gate 的判决不是“谁的绝对毫秒数更漂亮”，而是以下可证伪分流。

对出现 S0 full recompute miss 的 cell，定义非排队恢复服务时间
`T_service = R1 TTFT - R1 queue delay`，并定义
`Δservice(L, M) = median(T_service of S0 full-miss runs at (L, M)) - median(T_service of S0 APC-hit runs at (L, low-M))`。
它只用于排除“miss 存在但恢复服务成本没有实质头寸”的 cell；不把跨 cell 的差异解释为
ToolGap 收益。

`Δservice` 是跨 M 的**净非排队服务差**，不是纯 recompute-cost 估计：在启用
chunked-prefill 时，高压力 cell 的 prefill/decode 交织也可能进入该差值。每个 material
判断还必须有 S0 的 cached/recomputed-token accounting，证明缺失前缀确实触发 recompute；
时间与 token accounting 不一致时，经济头寸为 `inconclusive`，不能仅凭 `Δservice` 进入
A1。

### Stop / narrow：stock 已覆盖声明工作负载

满足任一情况即不进入 A1 performance-runtime 主线：

1. 在所有 target/overload cell，S0 的 R1 缓存 token 已达到对应 qualified anchor 给出的
   完整-prefix ceiling：没有可观察的 GPU recovery miss；或
2. 所有出现 full S0 recompute miss 的 cell 均满足 `Δservice(L, M) ≤ θ`，其中
   `θ = max(5 ms, 5% × low-M median service time)`；这种可重复但经济上无实质头寸的 miss
   不得计入 Continue 依据。`θ` 是 A0.2 screening threshold，不是生产 SLO 或通用用户感知
   阈值；或
3. 对每个 S0 的 GPU recovery miss，S1 都有与缺失 prefix 对应的 CPU restore 证据，且
   `transfer_overlap_observable=true` 且没有出现下文定义的方向性 trade-off；或
4. S0 或 S1 在所有 target/overload cell 对 foreground resume 与 active-probe tail 都
   同方向不劣，未出现 action preference reversal。此时最强诚实结论是一个 stock static
   配置足够，不能为 T 预设性能价值。

这里“方向性 trade-off”定义为同一 cell 的五个配对中至少四对显示：一个 policy 的 R1
TTFT 更低、另一个 policy 的 active-probe **median** queue-plus-TTFT 更低。cell-level
p95/p99 作为尾部 guardrail 单独报告，不能由每 run 的少量样本伪造。该规则用于识别稳定
冲突，不是统计显著性或性能优越性声明。

### Continue to A1：只记录 candidate-addressable gap，未声称 T 会赢

仅当下列任一条件在满足 `Δservice > θ`、且 token accounting 证明发生 missing-prefix
recompute 的 target/overload material cell 中成立，才允许测试 A1 的外置 Controller seam：

1. S0 的可复用前缀低于对应 qualified anchor 上限，且 S1 没有可归因的 CPU restore（或 CPU tier 已按
   本规格充足仍发生 capacity/load failure）；
2. 在至少两个预注册 material cell 的五个配对中至少四对出现上述方向性 trade-off；或
3. S0/S1 的 foreground resume 方向在两个以上预注册 material cell 发生 reversal，且每个方向在
   对应 cell 的至少四对中重复出现。

这只证明 stock 的静态路径存在待解释的 recovery/全局代价边界。A1 仍必须证明
candidate-owned controller 能通过合法 request-scoped seam 改变行为；不能把 A0.2
的差异归因给 ToolGap。

### Inconclusive / invalid

- 五对样本无稳定方向；
- cache/CPU-hit 证据不可归因；
- restore/probe decode overlap 不可见，却观察到 tail 差异；
- `Δservice` 与 cached/recomputed-token accounting 对经济头寸给出冲突信号；
- CPU tier、provenance、working set 或 active probe precondition 不满足；
- 输出 oracle 失败；
- engine 重启、环境、模板或 model revision 不一致。

这时保留原始证据并修复实验，不以“可能有 gap”为理由进入 A1。

## 9. 与后续阶段的边界

| 本规格给出的证据 | 不能给出的证据 |
| --- | --- |
| 固定环境中 stock APC/native offload 的可用性、miss/restore 归因与初步全局 trade-off | ToolGap runtime 已实现或更快 |
| 是否有理由测试 A1 外置 lifecycle seam | per-lifecycle retain、epoch/fallback/cancel correctness |
| 受控 capacity-pressure barrier 下的 stock 边界 | 真实 tool-gap 时间驱动 LRU、真实 tool latency 分布、生产代表性、CT3 break-even、CT4 policy value |

若进入 A1，本规格及所有 A0.2 run 都仍只是 stock baseline；A1 不得替换其 S0/S1 comparator。
