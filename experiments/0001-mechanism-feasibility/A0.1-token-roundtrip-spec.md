# Gate A0.1：Canonical Tool-Call Token Round-Trip 规格

> 状态：`roadmap`
>
> 决策用途：在任何 ToolGapController、custom wrapper、GPU APC/offload 比较之前，
> 判定真实工具调用的 canonical history 能否保留可复用的精确 token 前缀。

## 1. 范围与非目标

本规格只回答 token compatibility：模型生成的 tool call 经 vLLM parser 形成
canonical assistant message、追加 tool result 并重新提交后，R0 与 R1 是否保留足够
长的精确 token 前缀。

它不测 GPU hit、CPU restore、HBM 压力、性能、Controller 行为或 tool-choice 质量。
任意通过结论只适用于下述固定引擎、模型 revision、tokenizer/template、parser 和
fixture；不构成普适性或性能结论。

本 gate 只覆盖**单轮、单个 tool call**。多轮 tool-call 的累积 canonicalization 漂移
不是本 gate 的隐含结论，留待后续独立规格。

## 2. 固定组合与执行前 pin

| 项目 | 规定 |
| --- | --- |
| vLLM | `v0.25.1` / `752a3a504485790a2e8491cacbb35c137339ad34` |
| 模型 | `Qwen/Qwen2.5-7B-Instruct` |
| 权重与 tokenizer | 首次运行前解析为同一 Hugging Face commit SHA，并写入 manifest |
| chat template | 该 SHA 的 `tokenizer_config.json` 内置 template；保存原文与 SHA-256；不得传自定义 template |
| tool parser | `hermes` |
| 解码 | 单工具、非流式、`temperature=0` |
| tool-call trigger | 工具列表恰有一个固定 schema；R0 的 Hermes parser 必须实际解析出**恰好一个** tool call |
| 调度相关开关 | 显式 resolved 的 chunked-prefill；关闭 speculative decoding；写入 manifest |

本 gate 不使用 Chat Completions 的 `tool_choice='required'` constrained decoding；该能力不在
进程内 `LLM.chat()` 的 API 合同内。固定 fixture 是否触发单工具调用是一个可观察前置条件，
而非服务层保证：parser 不是恰好解析出一个 tool call 的 run 为 `invalid_run`。这不取代模型
实际生成的 tool-call tokens 或 Hermes parser。

在任一 R0/R1 配对前，先进行五次**新 engine 进程**的 R0 reproducibility preflight：
相同 fixture、固定启动参数、无背景请求下，五次 `RequestOutput` 的完整 R0 token-ID
数组必须逐 ID 相同。否则 fixture 不是可稳定锚定的输入，记为 `invalid_run` 并停止；
`temperature=0` 本身不是这个结论的替代证明。通过后，按预先记录的 ordinal 选择其中一条
R0 继续 R1 round-trip。该 ordinal 固定为第 5 次；前四次 engine 在采集 R0 后退出，第五次
engine 在采集 R0 后保持存活并立刻提交 R1。保存五条 preflight 的原始数组及比较结果。

## 3. R0、R1 与 canonical history

### R0：实际生成的 token 真值

```text
R0 = final RequestOutput.prompt_token_ids
   + final RequestOutput.outputs[0].token_ids
```

不得从可见 assistant 文本、structured tool call 或重新 tokenization 重建 R0。parser
输出只用于构造 R1 的 canonical assistant message。

### R1：实际 resume 请求的 token 真值

工具返回后，向同一 vLLM 进程提交一个新 request。其完整 canonical history 包含：

```text
原始 system/user messages
+ parser 输出的 assistant tool-call message
+ 具有相同 tool_call_id 的 role=tool result
+ 下一 generation prompt
```

R1 是该新 request 的 `final RequestOutput.prompt_token_ids`。本地
`apply_chat_template()` 仅用于标注 token spans；其输出必须与 R1 逐 token 完全一致，
否则为 `invalid_run`。

进程内 `LLM.chat()` 不构造 OpenAI Chat Completions response，因此它本身没有“实际返回的
`tool_call_id`”。在固定 commit 上，runner 只可调用
`vllm.entrypoints.chat_utils.make_tool_call_id()` 动态生成 ID；这是该版本 OpenAI serving 在将
parser `ToolCall` 包装为 response `ToolCall` 时使用的同一个 helper。它不得来自 fixture 或手写
静态常量；helper 的 module、symbol、vLLM commit 与生成值必须写入 manifest。tool result 使用固定
canonical JSON payload。

### Span 标注算法（预注册）

`a_start/a_end` 不是人工观察值，也不直接采信 parser 的结构化对象为 source map：Hermes
parser 未承诺保留模型输出的字符 offset。它们只在 `local render == R1` 后，由固定的
`a0.1_span_adapter` 算出；adapter 的版本/hash、template SHA、parser 名和 wire markers
写入 manifest。首次 R0 前，在已解析的 tokenizer/template SHA 上审阅 template 并冻结
`span_adapter.json` 的精确 `open_marker`、`close_marker` 与 non-nesting grammar；当前规格
不把任何 marker literal 预写成已验证事实。该文件 hash 一经写入 manifest 不得变更，随后
adapter 执行以下算法：

1. 对 R1 的 exact rendered UTF-8 text 按冻结 grammar 找到唯一、非嵌套的 `[open, close)` tool-call wire
   envelope；固定 system/user messages、tool schema 与 tool result 必须预检不含这两个 marker，
   以排除从其他 role 误匹配。
2. 只对**完整** rendered text 以同一 fast tokenizer、`add_special_tokens=False`、
   `return_offsets_mapping=True` 编码。其完整 token IDs 必须逐 ID 等于实际 R1；否则 local
   renderer 不是引擎输入真值，标为 `invalid_run`。不得对 prefix/envelope/suffix 分段编码后
   再以拼接相等作为判据。
3. 以 tokenizer 返回的完整序列 character offset mapping，将 `[open, close)` 映射为最小的
   连续 token 区间 `[a_start, a_end)`：`a_start` 是第一个与 envelope 有非空 character overlap
   的 token index，`a_end` 是最后一个有 overlap 的 token index 加一。若没有任何 token overlap、
   offset mapping 不可用或 offset 与 rendered text 不一致，标为 `invalid_run`。若边界 token
   同时覆盖 envelope 外字符，记录 `left/right_boundary_expansion`；token 是不可拆分的 APC
   单位，因此扩张后的完整 token 仍属于 span，不得因此虚假 `invalid`。
   该 span 含 tool-call wire wrapper 与 payload；assistant role header、template terminator 与
   tool role 仅在不共享边界 token 时位于 span 外。
4. 对 R0 output IDs 以 `skip_special_tokens=False`、关闭 cleanup 的 decoder 得到标注用 text，
   再按同一**完整序列 offset-mapping**算法定位唯一 envelope，得到 `r0_a_start/r0_a_end`；完整
   编码必须逐 ID 复原实际 generated IDs。R0/R1 envelope 内的 tool name 与 arguments 必须分别
   与 Hermes parser 的结构化结果一致。

因此，marker 缺失/重复/嵌套、完整序列 tokenization 不能复原、offset mapping 无法覆盖 marker，
或 parser/adapter 的**结构化**交叉校验失败，都是 span 无法可靠标注的 `invalid_run`，而不是由人
决定边界。结构化校验通过后，R0/R1 span 的逐 ID 差异是待测 token 事实，必须交由下文的
`semantic_span_equal` 判为 `serialization_stop`，不得被 adapter 吞为 `invalid_run`。

## 4. Mismatch 定位与可复用 block

R0/R1 的 canonical assistant tool-call 语义 token 是否逐 ID 一致是**待测假设**，不是
前提。不得预先断言首个差异只会是 terminator。driver 必须以 token-ID LCP 定位首个差异，
并报告其在 R1 坐标中属于 `before_assistant_semantic`、`assistant_semantic` 或
`after_assistant_semantic_or_r0_exhausted`；若 R0/R1 已在该位置结束，也必须显式记录缺失侧，
而不是伪造一个 token mismatch。

`after_assistant_semantic_or_r0_exhausted` 是 template-control / tool-role 诊断窗口，其中包含
terminator。该细分不影响判决：只要完整可复用 block 已覆盖 `a_end`，之后的差异不阻止 pass；
否则仍不足以支持 A0.2。

令：

```text
B        = 实际 vLLM cache block_size
a_start  = R1 内 canonical assistant tool-call token span 的起点（inclusive）
a_end    = R1 内 canonical assistant tool-call token span 的终点（exclusive）
L        = LCP(R0, R1)，以 token IDs 计算
reusable_full_block_ceiling = B * floor(L / B)
```

`a_start/a_end` 是 R1 坐标；artifact 同时必须记录 R0 侧对应 assistant semantic span
`r0_a_start/r0_a_end`，以及两段切片的逐 ID 对齐结果。system/user 的天然公共前缀不能
单独构成 token compatibility 证据。

```text
semantic_span_equal = R0[r0_a_start:r0_a_end] == R1[a_start:a_end]
```

`reusable_full_block_ceiling` 是 A0.2 唯一允许消费的完整-block 可复用上限。它必须覆盖
整个 assistant tool-call semantic span，即 `reusable_full_block_ceiling >= a_end`；仅覆盖
`a_start` 或一个语义片段都不够。任何 control-token 差异都会自然缩短 `L`，不会被计作
APC 命中。

## 5. 判决

| 状态 | 条件 | 后续动作 |
| --- | --- | --- |
| `pass` | R0 reproducibility preflight 通过；parser 恰好一个 tool call；R0/R1 均为引擎真值；local render 等于 R1；`semantic_span_equal`；`reusable_full_block_ceiling >= a_end` | 允许准备 A0.2 stock-sufficiency manifest。 |
| `serialization_stop` | assistant semantic span 内出现 token 差异，或 semantic span 虽相等但 `reusable_full_block_ceiling < a_end` | 保存 mismatch 位置与 block coverage；停止 ToolGap-KV runtime 主线或转为 canonical serialization compatibility 评审。 |
| `invalid_run` | R0 reproducibility preflight 失败；parser 数量不为一、采集点不是引擎真值、local renderer 不等于 R1，或 span 无法可靠标注 | 修复 harness 或按预先规定的 fixture 长度重试；不得作为 APC/Controller 结论。 |

## 6. 每次 run 的不可变原始证据

```text
raw/a0.1/<run-id>/
  manifest.json
  span_adapter.json
  template.jinja
  r0.json
  r1.json
  verdict.json
```

- `manifest.json`：引擎 commit/patch、model/tokenizer SHA、template SHA、parser、完整
  启动参数、block size、GPU/driver/CUDA、fixture/tool schema hash。
- `span_adapter.json`：version/hash、template SHA、精确 wire markers、non-nesting grammar 与
  adapter 的边界验证规则。
- `template.jinja`：实际使用的内置 template 原文。
- `r0.json`：初始 messages、prompt/generated token IDs、原始输出、parser 结构化结果、
  实际 tool_call_id、R0 assistant semantic span。
- `r1.json`：canonical history、固定 tool result、实际 prompt token IDs、local-render
  token IDs 与逐 token 相等断言。
- `verdict.json`：R0/R1 两侧 assistant span、span 逐 ID 对齐结果、首个 mismatch 与其
  region、LCP、`reusable_full_block_ceiling`、`a_end_mod_block_size`、
  `coverage_slack = reusable_full_block_ceiling - a_end`、状态和理由。

完整 token ID 数组必须保存；可附加哈希用于完整性校验，但哈希不能替代原始数组。

## 7. 执行前阻塞项

执行前必须填写、但目前尚未产生的 provenance 是：Hub commit SHA、GPU/driver/CUDA、
完整启动命令和实际 `block_size`。读取 `block_size` 前，本规格中含 `B` 的不等式只是
参数化模板，不能执行或预先宣布 pass。B 固定后不得通过 padding、改写 schema 或调整
fixture 来凑 block 边界；`a_end_mod_block_size` 与 `coverage_slack` 是移交 A0.2 时必须
审查的病态边界证据。这些字段未齐全时，不得把本规格或任何 mock trace 表述为已完成的
integration 或实验验证。

## 8. 实现边界（Ponytail 收口）

本 gate 是一次性 falsification harness，不是可扩展实验平台。实现最多新增以下四个
受版本控制的文件：

```text
experiments/0001-mechanism-feasibility/a0.1-fixture.json
src/toolgap_kv/a01.py
scripts/run_a01.py
tests/test_a01.py
```

- `a0.1-fixture.json` 固定初始 messages、单一 tool schema、固定 tool result 与下一轮
  prompt；其 hash 写入每次 run 的 manifest。
- `a01.py` 只包含不依赖 vLLM 的 LCP、span 标注/校验、判决与 artifact 完整性逻辑。
- `run_a01.py` 是唯一的真实、**进程内** vLLM `LLM.chat()` 驱动，不启动 HTTP server。
  R0 只取已完成 `RequestOutput.prompt_token_ids` 与
  `RequestOutput.outputs[0].token_ids`；R1 只取已完成
  `RequestOutput.prompt_token_ids`。第一条 R0 只验证 `tools` 确实进入实际 prompt；五条 raw
  R0 token-ID 数组逐 ID 相等后，才由第五条 R0 通过 pin 后的 Hermes parser 产生恰好一个
  canonical tool call，并用同 commit 的 `make_tool_call_id()` 动态生成 role=tool 所需 ID。raw
  token fields、parser bridge 或该 helper 任一不可得即记录 `invalid_run` 并停止，不以 text
  retokenization 或 HTTP response 代替。
- `test_a01.py` 只覆盖纯逻辑和 artifact 完整性；不得以 fake vLLM server 或 mock observer
  声称 integration 已验证。

不得引入 `ServingPath`、factory、protocol layer、fake serving implementation、通用配置
resolver、A0.2 artifact reader、HTTP Chat Completions 观察器、`AsyncLLMEngine` 或
controller/runtime 代码。`src/toolgap_kv/phase0.py` 表达的是
后续 A1 lifecycle action/trace 合同，不是 A0.1 token 证据域，不得为“复用”而耦合进来。
artifact 写入使用 Python 标准库：目标 run-id 已存在即拒绝覆盖；仅在 verdict 与六份原始
证据齐全后以临时目录 rename 发布。该边界是 `roadmap`，不是已存在的实现或集成声明。
