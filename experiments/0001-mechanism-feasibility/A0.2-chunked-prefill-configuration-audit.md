# A0.2 入场配置审计：chunked prefill

> 状态：`roadmap`；这是 A0.2 的硬入场 gate，不是 A0.2 结果。
>
> 决议草案：A0.2 不再沿用 `enable_chunked_prefill=False`；应先以 pinned vLLM 的受支持
> 默认配置 `enable_chunked_prefill=True` 完成新的 admission preflight，再冻结 A0.2 pin。

## 1. 为什么需要独立审计

A0.1R 的三个 no-pressure ordinal 在显式
`enable_chunked_prefill=False` 下都得到 `admission_pass`。但引擎启动时打印：

```text
This model does not officially support disabling chunked prefill.
Disabling this manually may cause the engine to crash or produce incorrect outputs.
```

三次成功只能证明该固定单请求 canonical pair 在该配置下实际跑通，不能证明它在 A0.2 的
builder/active-probe 容量压力下仍有正确的 scheduler 行为或稳定 accounting。因此，不能把
“显式固定配置”误当成“受支持配置”。

## 2. 锁定源码事实

审计对象：vLLM commit `752a3a504485790a2e8491cacbb35c137339ad34`。

### 2.1 warning 的条件

`vllm/engine/arg_utils.py` 的
`EngineArgs._set_default_chunked_prefill_and_prefix_caching_args()`
（`vllm/engine/arg_utils.py:L2480-L2502`）：

```text
default_chunked_prefill = model_config.is_chunked_prefill_supported

if enable_chunked_prefill is None:
    enable_chunked_prefill = default_chunked_prefill
elif runner_type == "generate" and not enable_chunked_prefill
     and default_chunked_prefill:
    warn("This model does not officially support disabling chunked prefill...")
```

这说明 Qwen2.5-7B 的 generate runner 在该 pin 下默认支持并选择 chunked prefill；显式
关闭才会触发“不官方支持”的 warning。warning 不是 A0.1R harness 的可忽略日志。

### 2.2 cached-token accounting 的数据流

以下路径与 `enable_chunked_prefill` 的取值解耦：

```text
KVCacheManager.get_computed_blocks()
  -> Scheduler.schedule(): num_new_local_computed_tokens
  -> Request.prefill_stats.set(local, external)
  -> EngineCoreOutput.prefill_stats
  -> OutputProcessor: RequestState.num_cached_tokens
  -> RequestOutput.num_cached_tokens
```

源码证据：

- `vllm/v1/request.py:L177` 无条件初始化
  `Request.prefill_stats = PrefillStats()`；
- `vllm/v1/core/sched/scheduler.py:L724-L783` 从
  `get_computed_blocks()` 获得 local computed tokens，并调用 `prefill_stats.set()`；
- `vllm/v1/metrics/stats.py:L264-L273` 定义
  `num_cached_tokens = num_local_cached_tokens + num_external_cached_tokens`；
- `vllm/v1/engine/output_processor.py:L627-L633` 将该字段复制到 request state，
  `L350-L374` 再将其写入 `RequestOutput`。

所以切换为受支持的 chunked-prefill 配置不会在源码层面删除 A0.1R/A0.2 使用的公开
accounting 字段。

### 2.3 它仍会改变什么

chunked prefill 决定 scheduler 是否能按 token budget 将长 prefill 与其他请求交织；这正是
A0.2 的 builder/active-probe 压力工作负载会触及的路径。故不能拿
`enable_chunked_prefill=False` 的 A0.1R admission 结果，直接外推到 A0.2。

### 2.4 为什么 A0.1R 当初关闭、现在应迁移

A0.1/A0.1R 当初显式关闭 chunked prefill 是一个有意的**隔离选择**：希望在单请求
canonical pair 上排除 prefill interleaving，令 token/accounting 的首个引擎验证尽量小。
这不是“该模型需要关闭它”的功能前提。

现在迁移不是为了追逐更漂亮的数，而是因为 A0.2 的 builder/probe 本来就需要研究多请求
scheduler interaction；继续使用被 upstream 明确标记为 unsupported 的 `False`，会让
“隔离变量”的收益小于配置不正确的风险。S0/S1 使用同一个受支持的 `True` 配置并记录
resolved config，仍然控制 policy comparison 的调度变量。

## 3. 建议的冻结配置

| 项目 | A0.1R 已运行 | 建议的 A0.2 候选 pin | 原因 |
| --- | --- | --- | --- |
| prefix caching | `True` | `True` | admission object 不变 |
| chunked prefill | `False`，有 unsupported warning | `True` | 使用 pin 的受支持默认调度路径 |
| speculative decoding | `None` | `None` | 排除额外 token/accounting 变量 |
| connector | 无 | S0 无；S1 仅原生 CPU connector | 保持 A0.2 比较边界 |
| cached-token oracle | `RequestOutput.num_cached_tokens` | 同上 | 已有源码传播契约，仍须 runtime 复验 |

这是一个**候选** pin，不是已经生效的 A0.2 规格。任何 A0.2 实现或 GPU comparative run 都
必须等待下节 preflight 通过。

## 4. 新 scheduler pin 的最小 admission preflight

不能修改或覆盖 A0.1R raw bundle；必须创建独立、可审计的 preflight artifact。

### 固定输入

- 继续使用 A0.1 fixture、同一模型/tokenizer revision、同一 vLLM commit、同一
  192-token anchor `C`；
- 使用本地、指定 revision 的模型 snapshot，禁止在 run 中依赖 Hub 出口；
- `enable_prefix_caching=True`、`enable_chunked_prefill=True`、无 connector、无 speculative
  decoding、单请求、无 builder/probe；
- 每个 ordinal 均为 fresh in-process `LLM`，R0 在隔离态完成，再立即提交 canonical R1。

### 判据与次数

注册三个 ordinal；每条均需：

```text
resolved scheduler_config.enable_chunked_prefill == True
R0.num_cached_tokens == 0
R1.num_cached_tokens == C == 192
LCP == 199
semantic_span_equal == true
```

这里的 `R1.cached == C` 不是由“字段存在”推出的先验结论，而是新 scheduler pin 必须
**runtime 验证的 accounting-contract hypothesis**。它必须保持精确，因为 A0.2 的
stock-hit/miss 判决要继续消费同一个冻结 anchor `C=192`。

不能把它改写为看似更宽松的
`block-aligned && C <= cached <= LCP`：在本实验的 `B=16`、`C=192`、`LCP=199` 下，该
谓词的唯一整数解仍是 `192`。若未来引擎在 chunked path 报告不同值，不能把它自动视为
同一个 APC hit；它意味着公开 accounting mapping 或其边界语义改变，必须停止 A0.2，先
建立新的、经评审的 oracle。

任一 counter 缺失、pin 漂移、token/anchor 漂移或输出 oracle 失败为 `invalid_run`；任一
`R1.cached < C` 的 block-aligned shortfall 首先回到源码 mapping 审计，不把它直接叫作
stock APC 不可用。任何 `R1.cached != C`（包括非 block-aligned 或大于 `C`）都同样暂停
该 gate，标为 `accounting_contract_change`，而非 admission pass。三个 ordinal 均通过，才
冻结此 scheduler pin；不以多数投票或事后补跑替代。

### 该 preflight 不能证明

- A0.2 压力下的 miss、CPU restore 或 queue/TTFT；
- chunked prefill 在所有并发模式下正确；
- builder/active-probe 压力下 chunked interleaving 是否改变 per-request accounting；该
  风险只能由 A0.2 comparative run 的每次 mapping/anchor check 持续监控，不能由本单请求
  preflight 解除；
- ToolGap runtime 或 policy 的性能价值。

它只把 A0.2 的 baseline 从“未官方支持的显式配置”迁移到“同一 canonical pair 上已做
runtime admission 验证的受支持配置”。

## 5. 完成后的决策

| 结果 | 动作 |
| --- | --- |
| 3/3 admission pass | 更新 A0.2 spec 的 chunked-prefill pin，开始 A0.2 implementation/review |
| valid `R1.cached < C` | 停止 A0.2，复核 APC/accounting mapping 与 scheduler interaction |
| invalid/环境失败 | 修复 harness 或 provenance，再以新 attempt 留痕；不解释为 APC 结论 |

在这一 gate 关闭前，D027 只允许 source audit 和 preflight 设计，不允许 A0.2 comparative
matrix、offload baseline 或 lifecycle runtime 实现。
