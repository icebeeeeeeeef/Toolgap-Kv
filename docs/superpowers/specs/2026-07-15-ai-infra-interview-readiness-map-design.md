# AI Infra 面试就绪知识地图设计

> 状态：已确认的组织基线
>
> 日期：2026-07-15
>
> 目标岗位：2027 校招 LLM Serving / 大模型推理平台 / AI Infra 平台研发 / 推理调度 / 模型服务后端

## 1. 目标

在完成并能够深挖 ToolGap-KV 的同时，建立一份可验收的面试知识清单，覆盖：

- 基础知识口述与连续追问；
- 真实系统源码、数据流、控制流与设计取舍；
- 非 LeetCode 领域手撕与本地测试；
- LeetCode 类算法题；
- 可复现实验、性能分析与失败边界；
- ToolGap-KV 的真实实现证据与项目深挖。

最终产物是一个浏览器可直接打开的 HTML 思维导图。节点可以展开或收起，并展示目标熟练度、验收标准、项目关联、优先级和学习状态。

## 2. 组织原则

采用“主题树 + 能力维度 + 独立详情页”，不使用单一的笼统熟练度，也不把 API、公式和练习全部画成导图节点。

主题主干暂定为：

1. 模型与推理原理
2. PyTorch 与张量编程
3. LLM 推理引擎与调度
4. KV Cache 与内存系统
5. 性能分析与实验方法（包含 GPU 最小认知）
6. 分布式推理与通信
7. 系统与语言基础
8. 领域手撕
9. LeetCode 算法
10. ToolGap-KV 项目深挖

主题之间允许交叉引用，但每个知识点只有一个主要归属，避免重复统计学习进度。

只有同时具有独立优先级、目标熟练度、完成状态或复习决策的内容，才能成为导图节点。API、公式、子概念、练习题、追问题和资料链接保存在对应主题详情页中。

最终 HTML 使用渐进式披露：导图节点默认只展示主题摘要；点击后在详情面板中展示完整知识列表。因为产物是本地单文件，详情数据可以嵌入 HTML，不要求通过网络按需加载。

## 3. 熟练度模型

每个节点分别评估四个能力维度：

- **原理**：能否从第一性原理解释机制、复杂度和资源约束，并承受连续追问。
- **源码**：能否定位真实系统实现，解释核心对象、数据流、控制流、并发语义和 trade-off。
- **手写**：能否在限定时间内完成简化实现、说明删减边界并通过本地测试。
- **实验**：能否设计并运行复现实验，正确解释指标、归因、噪声、负面结果和适用边界。

每个维度使用四级目标：

| 等级 | 可验收定义 |
|---|---|
| L0 了解 | 该维度不要求形成独立能力；知道用途和边界即可，不单独阻塞节点完成 |
| L1 能讲 | 脱稿讲清基本原理、输入输出、主要复杂度和典型用途 |
| L2 能深挖 | 能解释源码路径、关键数据结构、设计取舍、失败模式和替代方案 |
| L3 能实现 | 能独立手写或修改实现，完成调试、本地测试或可复现实验 |

L3 不要求所有主题都“从零重写生产系统”。每个主题必须单独定义实现范围和时间限制。

### 源码与复现触发协议

知识地图不要求每个概念都精读生产源码。一个特性满足以下任一门槛时，详情页必须增加“源码与复现契约”：

1. **真题证据门槛**：已收集真题明确询问框架源码、关键数据结构、执行流程、框架改造或领域手写；
2. **核心模块门槛**：即使真题只问原理，该特性仍是目标岗位的核心机制，仅靠概念无法解释真实系统行为、性能边界或工程取舍。

这里的“特性”指需要映射到生产实现的 Serving 机制。语言基础、硬件常识和实验方法等支撑节点不会仅因为优先级是 P0 就自动触发生产源码复现；它们按自身目标维度验收。

触发后采用“单一主实现 + 条件式对比实现”，不为关键词覆盖同时精读多个框架：

- **主实现**：选择与目标岗位、当前项目和真题证据最相关的一个主流项目，固定 release/commit 后读透；
- **对比实现**：只有设计存在实质差异、目标 JD 明确要求，或比较能回答一个具体 trade-off 时才加入；
- **机制复现**：从空文件实现删除生产复杂度后的核心 contract，并用本地测试验证状态、不变量和失败路径；
- **真实映射**：运行或 trace 固定版本真实框架，证明 toy 模型能够映射到生产对象、控制流、数据流和观测指标。

只有 toy 实现不能证明能够阅读生产系统；只运行官方示例也不能证明理解内部机制。两层证据都完成后，才满足核心模块的复现要求。

当一个相邻主题明确将手写设为 L0，并且独立重写的主要内容属于目标岗位之外的 CUDA/Triton kernel、硬件算子或模型训练时，可以使用“完整参考实验 + 生产集成路径 trace”替代从空文件机制重写。该例外必须在详情页说明岗位依据、实验中的主动修改、真实 dispatch/fallback 和依赖边界；它不能用于降低 KV Cache 生命周期、调度、并发状态机或其他目标岗位核心控制流模块的 toy 实现要求，也不会因此获得手写能力等级。

每份契约至少记录：

```text
触发依据：真题证据 / 核心模块判断
主实现锚点：项目 + 固定 release/commit
源码深度：入口、核心对象、控制流、数据流、不变量、trade-off
机制复现：toy contract、时间限制、本地测试和失败路径
真实映射：运行命令、trace/指标、原始结果和适用边界
对比实现：可选项目与明确的比较问题
明确不要求：不符合目标岗位收益的底层细节或完整重写
完成证据：源码路径图、代码、测试、命令、结果和负面案例
```

对非算子/GPU kernel 岗位，理解 kernel 接口、布局、复杂度和性能瓶颈通常是必要边界；只有真题、JD 或项目实现确实要求时，才把 CUDA/Triton kernel 手写升级为必修。

GPU 知识不单独扩张成算子学习主线。对当前目标岗位，它默认归入“性能分析与实验方法”，只要求能够解释 Serving 行为和完成可信测量。SM/warp、HBM、异步执行和数据搬运只保留系统直觉；CUDA/Triton kernel、PTX/SASS、occupancy 调优和 Tensor Core 指令不进入 P0。

## 4. 节点记录模板

主导图中的每个知识节点至少记录：

```text
主题名称
岗位权重：P0 / P1 / P2 / P3
目标熟练度：原理 L? / 源码 L? / 手写 L? / 实验 L?
一句话能力结果
知识边界摘要
关联主题
详情页路径
当前状态
```

对应详情页记录完整知识列表、手写题、实验、源码锚点、源码与复现契约、常见错误、原理追问和完成证据。详情内容不自动转换成导图子节点。

优先级定义：

- **P0**：不掌握就不应开始目标岗位面试。
- **P1**：高频或能够明显提高通过率，应在集中投递前完成。
- **P2**：岗位相关但依赖具体面试官或 JD，可在投递期间补齐。
- **P3**：超出目标岗位主线，作为防守性知识或未来扩展。

一个独立 P1/P2 节点可以在详情页声明紧凑的“P0 防守子集”，用于避免面试启动时留下明显概念盲区。防守子集只要求其中明确列出的口述或边界能力，不把整个节点升级为 P0，也不要求提前完成该节点声明的全部源码、手写和实验等级。最终 HTML 应分别展示节点优先级与防守子集，不能把两者合并成同一个完成状态。

## 5. 单主题讨论流程

每个主题按以下顺序讨论：

1. 说明它对目标岗位和 ToolGap-KV 的真实价值；
2. 划定需要掌握和明确不需要掌握的边界；
3. 拆出不过细但可验收的二级知识节点；
4. 为每个节点设置四维目标等级；
5. 检查真题证据和核心模块两个门槛，必要时确定主实现、对比实现和双层复现；
6. 给出面试追问、手写题和实验任务；
7. 用对抗式审查检查是否过浅、过宽或与岗位错位；
8. 用户确认后写入持续维护的知识地图 Markdown。

讨论顺序不等于最终思维导图的目录顺序。优先讨论能够支撑项目实施、又是面试高频门槛的主题。

## 6. ToolGap-KV 边界

ToolGap-KV 当前材料中的 `roadmap`、`shipped`、`experimentally validated` 和 `simulated` 必须保持原义。

知识地图不得因为存在项目设计文档，就把下列能力视为已经掌握：

- 真实 vLLM 接入和源码修改；
- 候选人自有生命周期控制器；
- GPU 性能测量；
- 故障注入、清理和恢复证据；
- 定量 break-even 或尾延迟结论。

项目完成证据与个人学习证据分别记录，然后在面试深挖节点中建立关联。

## 7. 输出文件

计划维护三类产物：

1. Markdown 主题索引：思维导图结构的权威内容源；
2. Markdown 主题详情：节点点击后展示的完整知识清单；
3. HTML：从已确认 Markdown 汇总生成的最终交互式思维导图。

```text
docs/interview-prep/
├── AI_INFRA_INTERVIEW_READINESS.md
└── topics/
    └── NN-topic-name.md
```

HTML 至少支持：

- 节点展开与收起；
- 关键词搜索；
- 按优先级、目标熟练度和完成状态筛选；
- 节点详情展示；
- 主图默认不渲染 API、公式、练习和追问等详情项；
- ToolGap-KV 关联节点和独立面试准备节点的视觉区分；
- 本地直接打开，不依赖在线服务。

## 8. 完成条件

只有同时满足以下条件，才称为“可以开始面试”：

- 所有 P0 节点达到各自目标等级并留下验收证据；
- P0 领域手撕和算法题能够在限制时间内通过本地测试；
- ToolGap-KV 的简历表述只使用已有证据支持的 claim state；
- 至少完成一次覆盖基础知识、领域手撕、算法题和项目深挖的综合模拟面试；
- 对未完成的 P1/P2/P3 节点能够明确说明风险，而不是把它们误判为已掌握。
