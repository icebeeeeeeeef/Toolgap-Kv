# vLLM v0.25.1 Gate A 源码审计

> 审计日期：2026-07-18
> 上游对象：`v0.25.1` / `752a3a504485790a2e8491cacbb35c137339ad34`
> 结论状态：`roadmap`。这是对固定源码和上游测试的审计，不是 ToolGap-KV
> 已集成、已运行或已测量的证据。

## 结论

**接受该 commit 作为 Gate A Week 1 的唯一源码审计基线，并允许它进入一个小型
Week 2 vertical slice；不宣告 Gate A 通过。**

这不是“vLLM 已有 tool-gap runtime”的结论。相反，源码证明了两个同时成立的事实：

1. 它已经有一条可复用的物理 KV 路径：v1 KV connector、CPU tier 的异步
   store/load、完成回传、load failure 到 recompute/fail、取消与延迟释放。
2. 它没有 tool-call waiting 或 tool-result continuation 的逻辑生命周期语义。
   一段工具调用结束后，恢复只能以一个新的 vLLM request 进入；`session/turn/epoch`
   也不是 vLLM 的一等字段。

因此最小可维护假设不是改写 scheduler，也不是把 connector 当作完整 runtime：

```text
应用内 ToolGapController
  维护 (session_id, turn_id, epoch) 和合法转移
  将受控 resume 的 lifecycle envelope 写入 SamplingParams.extra_args
        │
        ▼
固定 vLLM v0.25.1
  外置 ToolGap OffloadingSpec/Manager（或窄 connector wrapper）
  读取 kv_transfer_params，委托 vLLM 的 CPU tier 与 GPU block 生命周期
```

这里的 controller 必须以行为而非 trace 证明存在：移除它时，受控 request 不再携带
envelope、不再走被选择的 lifecycle 分支；普通请求继续委托 vLLM 默认路径。

## 推荐的 Gate A 最小实现边界

1. 只启用 v1 `OffloadingConnector` 的单 CPU tier；不使用分布式 connector、二级 tier、
   LMCache 或 `main`。
2. 新增一个外置 `ToolGapCPUOffloadingSpec`，其 manager 组合内置 CPU manager；它只读取
   私有 `toolgap` envelope、维护逻辑 claim/epoch、决定是否允许该受控 request 的
   store/load，并输出 DecisionTrace 事件。
3. tool 调用和外部 handler 由同一 Python driver 编排。handler 回来后，以**新的** request
   提交完整的 canonical conversation（含 assistant tool-call 与 tool result）；不伪造
   vLLM 内部 continuation。
4. Week 2 只跑单工具、串行、quiescent fixtures：
   - GPU 未 reset：验证相同前缀的 `gpu_hit`；
   - 仅 reset GPU prefix cache：验证 CPU load 发生；
   - reset GPU 与 connector cache：验证零可用 cache 下重算。
5. 这三个 fixture 的 global reset 仅是机制 conformance 手段。它们**不能**外推为并发
   agent 的 per-claim 驱逐策略或 CT3 性能边界。

## Gate A 的明确停止/升级条件

| Week 2 观察 | 决策 |
| --- | --- |
| 三条串行路径都可归因；custom manager 可让普通请求走默认分支；移除 controller 后受控分支消失 | 接受该版本进入 CT1，开始实现 lifecycle state machine 与 epoch tests。 |
| CPU restore / recompute 可工作，但 controller 只能记录、无法改变 store/load/恢复行为 | 停止：这是 benchmark adapter，不满足 candidate-owned runtime。 |
| 需要按单个 lifecycle claim 定向驱逐，且外置 spec/manager 无法在不触碰 private block state 的情况下实现 | 先写一个 unmodified-vLLM failing test 和最小缺失契约；只有该证据成立才允许一个小 core patch。 |
| GPU+CPU global reset 都无法稳定区分 CPU hit 与 recompute，或 completion 不能归因到正确 request | 拒绝此版本作为执行基线；记录 trace 后重新选择版本，而不是静默改 tag。 |

## 不应从本报告推导的结论

- 不代表 v0.25.1 已满足 Gate A pass conditions；没有本项目的 GPU trace、controller、
  removal test 或 closed decision card。
- 不代表 `retain` 已是安全的 per-session HBM 保留能力；vLLM prefix cache 是共享物理状态。
- 不代表全局 reset 可用于多 agent / 连续 batching 情景。
- 不代表 `kv_transfer_params` 是长期稳定 API；`KVConnectorBase_V1` 与
  `max_offload_tokens` 明确标为 experimental，故所有依赖点必须锁在本 commit、测试并
  记录 patch hash。([base.py](https://github.com/vllm-project/vllm/blob/752a3a504485790a2e8491cacbb35c137339ad34/vllm/distributed/kv_transfer/kv_connector/v1/base.py#L180-L200), [offloading scheduler.py](https://github.com/vllm-project/vllm/blob/752a3a504485790a2e8491cacbb35c137339ad34/vllm/distributed/kv_transfer/kv_connector/v1/offloading/scheduler.py#L250-L268))

## 下一步的唯一研发问题

不是继续横向比较版本，而是用上述固定 commit 回答：

> 一个 custom offloading manager 能否在不直接操作共享 GPU block/refcount 的前提下，
> 对携带 `toolgap` envelope 的新 request 实施一个真实的 `offload` 或 fallback transition，
> 并用 epoch 拒绝 stale completion？

这个问题若得到实证“是”，项目才有 CT1 runtime 主线；若为“否”，我们得到的是
精确的缺失契约，而不是含糊的“vLLM 不好改”。
